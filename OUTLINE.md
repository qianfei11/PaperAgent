# 图形化LLM交互工具开发大纲

## 项目概述

本项目旨在开发一个图形化工具，用于改善与大语言模型（LLM）的交互体验。通过构建上下文管理系统和文档关联库，解决LLM在长对话过程中出现的上下文不一致问题，特别适用于学术论文阅读和思考辅助。

## 核心功能模块

### 1. 上下文管理系统
- **会话摘要机制**：每次对话结束后自动提取关键信息
- **JSON大纲生成**：将摘要内容组织为结构化的JSON格式
- **上下文继承**：后续对话以JSON大纲为基础进行推理
- **动态更新机制**：根据新的对话内容更新现有大纲

### 2. 文档关联库
- **键值对存储系统**：维护关键词到文档路径的映射
- **文档类型支持**：支持PDF、JPG等常见学术文档格式
- **智能检索**：当LLM提及特定关键词时自动关联相应文档
- **文档预览**：集成轻量级文档查看器

### 3. 用户界面
- **对话面板**：与LLM交互的主要区域
- **大纲展示**：可视化显示当前会话大纲结构
- **文档库管理**：可视化管理文档关联关系
- **历史记录**：保存和回溯之前的对话会话

### 4. 数据管理
- **JSON大纲格式**：定义标准化的上下文存储格式
- **文档索引系统**：高效管理大量学术文档
- **会话持久化**：保存和恢复对话状态

## 上下文管理系统架构设计

### 系统组件构成

#### 1. 会话监听器（Session Listener）
- **职责**：监控LLM对话流，识别对话结束事件
- **实现**：WebSocket连接状态监控或API响应解析
- **触发条件**：LLM回复完成、用户明确结束对话或超时

#### 2. 内容摘要引擎（Content Summarization Engine）
- **职责**：分析对话历史，提取关键信息点
- **算法**：
  - 关键词提取算法（TF-IDF、TextRank等）
  - 主题建模（LDA或其他主题模型）
  - 句子重要性评估（基于位置、关键词密度等）
- **输出**：结构化摘要数据

#### 3. 大纲生成器（Outline Generator）
- **职责**：将摘要内容转化为JSON大纲格式
- **规则引擎**：基于对话内容类型自动分类（如研究背景、方法、结果等）
- **层次结构**：支持多层级大纲组织

#### 4. 上下文注入器（Context Injector）
- **职责**：在新对话开始前注入现有大纲作为上下文
- **策略**：根据对话相关性选择性注入部分大纲内容
- **大小控制**：确保注入内容不超过LLM的最大上下文长度

#### 5. 大纲更新器（Outline Updater）
- **职责**：根据新对话内容更新现有大纲
- **变更检测**：识别新增概念、修正观点、补充细节
- **一致性检查**：确保更新后的大纲保持逻辑一致性

### 数据流转流程

1. **对话监控**：监听器持续监控LLM交互过程
2. **对话结束判断**：识别对话自然结束或用户主动终止
3. **内容分析**：摘要引擎分析整个对话历史
4. **大纲生成/更新**：基于分析结果创建新大纲或更新现有大纲
5. **上下文注入**：在下次对话开始前将大纲注入提示中
6. **循环迭代**：每次对话后重复上述流程

### 技术实现要点

#### 摘要算法实现
- 使用NLP技术识别对话中的实体、关系和重要概念
- 实现层次化摘要：总体摘要 + 细节要点分解
- 融合向量化表示和传统NLP方法提升准确性

#### 大纲结构优化
- 动态调整大纲层级深度
- 支持跨话题关联和引用
- 提供大纲可视化编辑接口

#### 性能优化
- 缓存机制减少重复计算
- 异步处理避免阻塞用户交互
- 摘要压缩算法控制上下文大小

### 扩展性设计

- **插件化架构**：支持不同摘要算法的替换和扩展
- **配置化规则**：允许用户自定义大纲生成规则
- **API接口**：提供外部系统集成能力


## 文档关联库系统架构设计

### 系统组件构成

#### 1. 文档管理器（Document Manager）
- **职责**：负责文档的上传、存储、检索和删除
- **支持格式**：PDF、JPG、PNG、DOCX、TXT等学术常用格式
- **存储策略**：本地存储与元数据分离，避免占用过多内存
- **索引系统**：为文档建立快速检索索引

#### 2. 键值映射引擎（Key-Value Mapping Engine）
- **职责**：维护关键词到文档路径的映射关系
- **数据结构**：哈希表 + 倒排索引，支持快速查找
- **映射规则**：
  - 手动映射：用户手动指定关键词与文档的对应关系
  - 自动映射：基于文档内容自动提取关键词并建立映射
- **冲突处理**：处理同一关键词指向多个文档的情况

#### 3. 智能关联检测器（Intelligent Association Detector）
- **职责**：在LLM对话中检测关键词提及并自动关联文档
- **匹配算法**：
  - 精确匹配：完全匹配关键词
  - 模糊匹配：支持同义词、缩写等变体匹配
  - 语义匹配：基于向量化表示的语义相似度匹配
- **优先级机制**：根据匹配度和上下文相关性排序

#### 4. 文档预处理器（Document Preprocessor）
- **职责**：对上传文档进行预处理，提取可用信息
- **PDF处理**：文本提取、页面分割、表格识别
- **图像处理**：OCR识别、图像描述生成
- **文档解析**：提取文档标题、作者、摘要等元数据

#### 5. 文档检索服务（Document Retrieval Service）
- **职责**：根据关键词快速检索相关文档
- **缓存机制**：常用文档预加载到内存
- **异步加载**：大文档分块加载避免阻塞
- **权限控制**：确保文档访问安全

### 数据流转流程

1. **文档上传**：用户上传文档至系统
2. **预处理**：预处理器解析文档内容并提取元数据
3. **映射建立**：根据用户设置或自动分析建立关键词-文档映射
4. **对话监控**：检测LLM对话中的关键词提及
5. **文档关联**：自动检索并关联相关文档
6. **内容整合**：将文档内容整合到对话上下文中供LLM参考

### 技术实现要点

#### 存储方案
- **文档存储**：本地文件系统存储原始文档，数据库存储元数据
- **索引结构**：使用SQLite FTS或Elasticsearch实现全文检索
- **缓存策略**：Redis缓存频繁访问的文档和映射关系

#### 文档处理
- **PDF处理**：使用PDF.js或PyMuPDF进行文本提取
- **图像处理**：集成Tesseract OCR和图像识别模型
- **文档格式转换**：统一转换为文本格式便于LLM处理

#### 匹配算法
- **字符串匹配**：使用Trie树或AC自动机实现高效关键词匹配
- **语义匹配**：使用Sentence-BERT等模型计算语义相似度
- **权重分配**：根据匹配类型和置信度分配不同权重

### 扩展性设计

- **插件化格式支持**：可扩展支持更多文档格式
- **机器学习增强**：持续学习用户的映射偏好优化推荐
- **API接口**：提供外部系统集成能力
- **批量操作**：支持批量上传和映射操作

### 性能优化

- **懒加载**：仅在需要时加载文档内容
- **文档分块**：大文档分块处理避免内存溢出
- **并行处理**：多文档并行预处理提升效率
- **压缩存储**：文档内容压缩存储节省空间

```json
      "summary": "string",

      {
        "key": "string",

        "type": "string",
        "title": "string",

    "totalTokens": "integer",
    "totalMessages": "integer",
    "activeEntitiesCount": "integer",
    "documentsCount": "integer"
  }

```

### 数据结构说明

#### 1. 会话基本信息
- **sessionId**: 全局唯一标识符
- **title/description**: 会话标题和描述
- **createdAt/lastModified**: 时间戳记录
- **version**: 数据结构版本号

#### 2. 大纲结构（outline）
- **层次化设计**: 支持多层级嵌套结构
- **ID系统**: 每个节点有唯一ID，支持父子关系
- **内容摘要**: 每个节点包含标题、摘要和详细内容
- **关键点提取**: 自动提取的重要信息点
- **实体识别**: 识别出的关键实体词汇
- **标签系统**: 用户自定义标签便于分类
- **文档关联**: 与该节点相关的文档引用
- **置信度**: 系统对节点内容准确性的评估
- **时间戳**: 记录创建和修改时间

#### 3. 实体映射（entitiesMap）
- **实体中心化**: 以实体为核心组织相关信息
- **描述信息**: 实体的详细描述
- **文档链接**: 与实体相关的文档
- **出现记录**: 实体在哪些大纲节点中被提及

#### 4. 文档库（documentLibrary）
- **文档元数据**: 包含路径、类型、大小等信息
- **内容预览**: 文档内容的简短预览
- **实体关联**: 文档涉及的相关实体
- **类型支持**: 支持多种文档格式

#### 5. 对话历史（conversationHistory）
- **角色区分**: 区分用户和助手的消息
- **大纲关联**: 消息与大纲节点的关联关系
- **文档引用**: 消息中引用的文档

#### 6. 元数据（metadata）
- **统计信息**: 令牌数、消息数等统计数据
- **实体计数**: 活跃实体数量统计

### 扩展性设计

#### 1. 版本兼容性
- 使用版本号管理数据结构变更
- 支持向下兼容的数据迁移机制

#### 2. 扩展字段
- 预留扩展字段支持未来功能
- 使用通用metadata字段存储附加信息

#### 3. 性能优化
- 支持大纲的部分加载和增量更新
- 文档内容分块存储避免单次加载过大

### 数据验证规则

#### 1. 必需字段
- sessionId、title、outline至少包含一个根节点
- 时间戳格式必须符合ISO 8601标准

#### 2. 约束条件
- outline节点level不能超过预设最大层级
- entities数组长度限制防止过度膨胀
- relatedDocuments数量限制

#### 3. 数据完整性
- parentId必须对应存在的outline节点
- documentLibrary中的文档路径必须有效
- entitiesMap中的实体必须在outline中存在引用

这个数据结构设计既满足了上下文管理的需求，又支持文档关联功能，同时具备良好的扩展性和性能表现。


## LLM集成方案

### 集成架构设计

#### 1. 抽象接口层（LLM Adapter Interface）
- **职责**：定义统一的LLM交互接口
- **核心方法**：
  - `sendMessage(prompt: string, context: Context): Promise<Response>`
  - `streamMessage(prompt: string, context: Context, callback: Function): Promise<void>`
  - `getModelInfo(): Promise<ModelInfo>`
- **优势**：实现不同LLM提供商的无缝切换

#### 2. 上下文管理器（Context Manager）
- **职责**：将JSON大纲转换为LLM可理解的上下文格式
- **功能**：
  - 从JSON大纲提取关键信息
  - 控制上下文长度以适应LLM限制
  - 优先保留最近和最相关的上下文信息
- **策略**：
  - 基于token数量的截断策略
  - 基于重要性的内容筛选策略
  - 基于时间的新鲜度衰减策略

#### 3. 文档处理器（Document Processor）
- **职责**：将文档内容转换为LLM可处理的格式
- **功能**：
  - 文本提取（PDF、Word等格式）
  - 图像转文本（OCR、图像描述）
  - 内容摘要和关键信息提取
- **优化**：
  - 分块处理大文档
  - 基于相关性的内容筛选

### 集成实现细节

#### 1. 主流LLM提供商支持
- **OpenAI API**：GPT-3.5/4系列模型
  - 支持函数调用功能，便于上下文注入
  - 流式响应支持，提升用户体验
- **Anthropic API**：Claude系列模型
  - 更长的上下文窗口，适合处理复杂对话
  - 增强的事实准确性
- **本地模型**：Ollama、LocalAI等
  - 隐私保护，数据不出本地
  - 可定制化程度高

#### 2. 上下文注入策略
- **大纲摘要注入**：将JSON大纲的核心内容转换为自然语言摘要
- **实体列表注入**：提供当前讨论的实体清单
- **文档引用注入**：在适当时候注入相关文档内容
- **历史对话注入**：保留最近几轮关键对话

#### 3. 对话管理机制
- **请求构造**：
  ```
  System Prompt: [系统指令]
  Current Context: [JSON大纲摘要]
  Related Documents: [相关文档内容]
  Conversation History: [对话历史]
  User Query: [用户输入]
  Assistant Response:
  ```
- **响应处理**：
  - 解析LLM响应以提取结构化信息
  - 更新JSON大纲中的相关内容
  - 识别文档引用并建立关联
